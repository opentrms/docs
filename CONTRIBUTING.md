# Contributing to the OpenTRMS docs

## Information architecture

Docs are organized into five sections, each with a distinct job:

- **Overview** — what OpenTRMS is and why it's built this way (quickstart, principles, architecture, asset coverage).
- **Concepts** — how the platform works under the hood (event sourcing, the deal state machine, valuation, accounting, settlement, closeout, agents).
- **Guides** — task-oriented, step-by-step instructions (run locally, operate EOD, book a trade, drive via MCP, verify the audit trail).
- **Extend** — how to build on the platform (SPI contracts, writing a `ProductValuator`, the TCK, the new-product playbook, schema authoring, extensibility model).
- **Reference** — exhaustive, lookup-style material: a generated **API Reference** (`/reference/api/trms-api`, one page per endpoint) followed by the schema catalog, data dictionary, ERD, scopes, CLI, MCP tools, conventions, and cookbook.

When adding a page, put it in the section that matches its job, not the module it happens to describe.

## Maturity model

Every page opens with an eyebrow label (its section) followed immediately by a `<StatusBadge>`:

```mdx
<div className="eyebrow">Concepts</div>

<StatusBadge status="draft" reviewed="2026-06-23" />
```

Allowed `status` values:

| Status | Meaning |
| --- | --- |
| `stable` | Accurate, complete, safe to rely on. |
| `draft` | Has real content but is incomplete or unreviewed in places. |
| `proposal` | Describes an intended design, not yet implemented. |
| `planned` | Skeleton/stub only — content lands in a later phase. |
| `deprecated` | Superseded; kept for reference. |

`reviewed` is the date (`YYYY-MM-DD`) the content was last checked against the upstream source.

## Auto-generated pages — do not hand-edit

Three artifacts are pulled from upstream sources and regenerated by scripts. Hand edits to the generated regions will be overwritten:

- `docs/reference/erd.mdx` — the entire file is generated from `${OPENTRMS_BACKEND}/docs/ERD.mermaid`.
- `docs/reference/schema-catalog.md` — only the table between `{/* BEGIN GENERATED:schemas */}` and `{/* END GENERATED:schemas */}` is generated, from `${OPENTRMS_BACKEND}/schemas/**/*.json`. Prose above and below the markers is hand-authored and safe to edit.
- `docs/reference/api/**` — the entire **API Reference** section (one page per endpoint, plus tag landing pages and the introduction page) is generated by `docusaurus-plugin-openapi-docs` from the committed spec at `static/openapi/trms-openapi.json`. Every file under `docs/reference/api/` is generated output — never hand-edit it; edits will be silently overwritten the next time `npm run gen-api` runs. `docs/reference/api/sidebar.ts` is also generated and is required by `sidebars.ts` (`referenceSidebar`).

### Running the sync

```bash
npm run sync
```

By default the script reads from `~/ideas/opentrms` (backend) and `~/ideas/opentrms-workbench` (frontend, reserved for future generators). Override with environment variables if your checkouts live elsewhere:

```bash
OPENTRMS_BACKEND=/path/to/opentrms OPENTRMS_FRONTEND=/path/to/opentrms-workbench npm run sync
```

The script never fails the build: if a source file or directory is missing, it logs a warning and writes a clear placeholder instead of throwing.

### Refreshing the API reference

The API reference is generated from the committed OpenAPI spec at `static/openapi/trms-openapi.json`, not fetched at build time — this keeps `npm run build` reproducible offline (CI, no backend running).

To refresh both the spec and the generated pages in one step:

```bash
npm run refresh-api
```

This runs `npm run sync` (which also refreshes the spec — see below) followed by `npm run clean-api && npm run gen-api`.

The spec itself is obtained with this precedence, degrading gracefully (a missing source only logs a warning, it never fails the sync):

1. `OPENTRMS_OPENAPI` — path to a local spec file (JSON or YAML) to copy in as-is.
2. else fetch `TRMS_OPENAPI_URL` (default `http://localhost:8080/v3/api-docs`).
3. else leave the already-committed `static/openapi/trms-openapi.json` untouched.

```bash
# refresh from a locally running backend (default URL)
npm run refresh-api

# refresh from a non-default backend URL
TRMS_OPENAPI_URL=http://localhost:9090/v3/api-docs npm run refresh-api

# refresh from a spec file instead of a live server
OPENTRMS_OPENAPI=/path/to/trms-openapi.json npm run refresh-api
```

If you only need to regenerate pages from the spec already on disk (no fetch), run `npm run clean-api && npm run gen-api` directly.

## Before you submit

Run `npm run build` and confirm it completes with no broken links (`onBrokenLinks` is set to `throw`).
